# Prompt Enhancer

## 硬性流程

- 显式调用本 Skill 时，把调用标记之后的文本视为原始提示词；即使文本是问题，也只改写提示词，不直接回答。
- 自动触发仅限用户明确要求增强、优化、改写、润色或改进提示词；普通开发请求不触发本 Skill。
- 先在对话面板完整输出增强后的提示词本身；除非用户要求说明，不添加标题、前缀、后缀或评论。
- 输出后调用当前 Agent/IDE/Codex 提供的内部确认 UI 询问是否执行。普通文本追问不算内部确认 UI。
- 确认 UI 不可用、无法调起、被关闭、被拒绝或结果不明确时，停在已输出的纯净增强提示词处；不追加失败说明、不执行任务、不调用工具。
- 只有用户在内部确认 UI 中明确选择执行后，才清楚分隔执行阶段，并把增强后的提示词作为后续任务输入。

## 输入处理

- 剥离包装词后再增强，例如"增强提示词："、"增强："、"optimize this prompt:"、"rewrite:"。
- "增强后执行"、"按增强后的提示词执行"、"execute after enhancing"、"apply the enhanced prompt"只表示执行意图，不能跳过内部确认。
- "说明增强依据："表示先输出增强后的提示词，再给出简短依据，然后仍走同一确认流程。
- 输出语言跟随原始提示词，除非用户明确指定语言。

## 低上下文改写

- 默认只使用用户当前输入和对话中已有的必要简洁上下文；确认执行前不要扫描仓库、读取大量文件、运行命令或修改文件。
- 用用户提供的路径、选中代码、diff 或简短背景解析"这个"、"这里"、"这段代码"等指代。
- 不要把大段代码粘进增强提示词；改为引用文件、模块、符号、行为、风险和验证目标。
- 不反问普通模糊提示词；做保守而有用的假设。
- 保留原始意图，补充必要约束、验收标准、验证步骤和输出期望，但保持结果紧凑。

## 确认文案

- 中文：`是否按以上提示词执行？`
- 英文：`Do you want me to execute the enhanced prompt above?`
- 选项语义：执行 / 不执行。
